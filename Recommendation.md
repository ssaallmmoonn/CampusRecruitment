# 本系统的协同过滤算法开发步骤



把算法拆解开，其实就是**收集数据 -> 算数学题 -> 排序输出**的过程。

------

## 第一步：明确我们的“数据”是什么？（隐式反馈）

传统的协同过滤是用来做电影评分的（用户打 1~5 星），这叫**显式反馈**。但在招聘系统中，学生通常不会给岗位“打星”。 我们需要把学生的**行为**转化为**分数（隐式反馈）**。你可以这样定义权重：

- **浏览岗位**：1分
- **收藏岗位**：3分
- **投递简历**：5分

通过记录这些行为，我们就能在数据库中构建出一个**用户-岗位交互矩阵 (User-Item Interaction Matrix)**。



## 第二步：选择哪种协同过滤？

协同过滤主要分两类。对于你的毕设，我**强烈建议你采用“基于物品的协同过滤 (Item-based CF)”**。

- **为什么不选基于用户 (User-based CF)？**
    - 在招聘季，学生（用户）的兴趣变化极快，且用户数量可能远大于岗位数量，计算用户相似度的矩阵非常庞大且稀疏。
- **为什么选基于物品 (Item-based CF)？**
    - **稳定性高：** 一个“Java后端开发”的岗位和一个“数据库工程师”的岗位的相似性，是相对固定的，不会因为几个人投递了就发生剧烈变化。
    - **可解释性强：** 答辩时你可以这样解释你的系统：“因为该学生投递了『阿里前端开发』，系统计算发现『腾讯前端开发』与前者的历史被投递群体高度重合（相似度高），因此推荐了腾讯的岗位。” 这听起来非常有逻辑。
    - **性能更好：** 岗位库的数量通常比学生总数可控，计算物品相似度矩阵的性能开销较小，甚至可以每天凌晨跑一次定时任务离线计算。





## 第三步：采用什么公式？（论文核心素材）

在基于物品的协同过滤中，你需要解决两个数学问题。

### 1. 如何计算两个岗位的“相似度”？

**采用公式：余弦相似度 (Cosine Similarity)**

- **原因：** 余弦相似度衡量的是两个向量在方向上的夹角大小，而不太受向量长度（绝对数值大小）的影响。也就是说，哪怕“岗位A”是个几万人投递的热门大厂岗，“岗位B”是个几百人投递的初创公司岗，只要投递他们的人群画像高度重叠，余弦相似度依然能准确发现它们是相似的。

- **数学公式（记得写进论文里）：**

    设 $i$ 和 $j$ 是两个不同的岗位，$U(i)$ 是对岗位 $i$ 产生过行为的用户集合，$R_{u,i}$ 是用户 $u$ 对岗位 $i$ 的行为评分。岗位 $i$ 和 $j$ 的相似度 $sim(i, j)$ 为：

$$
sim(i, j) = \frac{\sum_{u \in U(i,j)} R_{u,i} R_{u,j}}{\sqrt{\sum_{u \in U(i)} R_{u,i}^2} \sqrt{\sum_{u \in U(j)} R_{u,j}^2}}
$$

### 2. 如何预测用户会对某个未看过的岗位打多少分？

**采用公式：加权求和预测 (Weighted Sum Prediction)**

- **原因：** 算出岗位间的相似度后，我们需要结合目标用户过去的行为，来预测他会不会喜欢某个新岗位。加权求和能够把用户曾经的高分行为（比如投递）放大，从而推测出他对相似岗位也会有高意向。

- **数学公式（记得写进论文里）：**

    设我们要预测用户 $u$ 对未接触过的岗位 $j$ 的评分 $P_{u,j}$。$N(j)$ 表示与岗位 $j$ 最相似的 $K$ 个岗位集合（并且是用户 $u$ 曾经产生过行为的）：

$$
P_{u,j} = \frac{\sum_{i \in N(j)} sim(i, j) \cdot R_{u,i}}{\sum_{i \in N(j)} |sim(i, j)|}
$$



（算出所有未接触岗位的预测得分 $P$ 后，按照得分从高到低排序，取前 10 名，这就是你要展示在前端 Vue 页面上的“为你推荐”列表。）



## 第四步：落地实现的避坑指南（非常重要）

即使算法选对了，在毕设答辩中，老师通常还会“刁难”你一个现实问题：**“如果你的系统刚上线，一个新注册的学生什么岗位都没点过（没有任何行为数据），你的算法怎么给他推荐？”**

这叫做**冷启动问题 (Cold Start)**。纯靠协同过滤是解不开这个死结的。 **你的对策（加分项）：采用“混合推荐策略”。**

1. **对于新用户（冷启动期）：** 当学生刚注册，简历没有投递记录时，采用**基于内容的推荐 (Content-based)**。用学生填写的“期望岗位、技术标签（如Vue, Java）”去直接匹配岗位表里的“要求标签”。
2. **对于老用户（活跃期）：** 当学生的“浏览+收藏+投递”记录超过一定数量（比如 5 条）后，系统自动切换为**基于物品的协同过滤推荐**。

在论文里把这个逻辑写上去，答辩老师会觉得你不仅懂算法，还真正懂软件工程的业务落地。





# 余弦相似度公式

## 1. 核心思想：它到底在算什么？

不要一开始就陷入数字中。想象一下，在空间里有两个带有箭头的线段（也就是“向量”）。

- **如果两个箭头指向完全相同的方向**，它们之间的夹角是 0 度，余弦值是 1。这代表它们**完全相似**。
- **如果两个箭头呈 90 度直角**，余弦值是 0。这代表它们**毫无关系**。
- **如果两个箭头方向完全相反**，余弦值是 -1。这代表它们**完全对立**。（在你的招聘系统中，因为用户的行为得分都是正数，比如浏览 1 分、投递 5 分，所以计算结果通常只在 0 到 1 之间）。

**一句话总结：余弦相似度看重的是两个人（或两个岗位）在“品味方向”上是否一致，而不是他们具体的“活跃程度（数值大小）”。**



## 2. 公式拆解（你的论文必写内容）

在你的系统中，你要计算的是“岗位 A”和“岗位 B”的相似度。数学公式如下：

$$sim(\mathbf{A}, \mathbf{B}) = \cos(\theta) = \frac{\mathbf{A} \cdot \mathbf{B}}{\|\mathbf{A}\| \|\mathbf{B}\|} = \frac{\sum_{i=1}^{n} A_i B_i}{\sqrt{\sum_{i=1}^{n} A_i^2} \sqrt{\sum_{i=1}^{n} B_i^2}}$$

看着唬人，其实分为上下两部分：

- **分子（内积 / 点乘）：$\sum_{i=1}^{n} A_i B_i$**
    - **物理意义：** 衡量这两个岗位在不同用户身上的重合度。只有当同一个用户对岗位 A 和岗位 B 都产生了行为（得分都不为 0），相乘才会有正数结果，累加起来才会变大。
- **分母（模长乘积）：$\sqrt{\sum_{i=1}^{n} A_i^2} \sqrt{\sum_{i=1}^{n} B_i^2}$**
    - **物理意义：** 起到“惩罚”或“标准化”的作用。如果岗位 A 是个几万人投递的“海王岗位”，它的数值天然就大。分母除以它自己的长度，就可以把它的影响力拉回到正常水平，防止热门岗位与所有岗位都产生高相似度的假象。



## 3. 沙盘推演：用你的招聘系统数据算一次

为了让你彻底明白，我们手算一个极简的例子。假设你的系统里只有 3 个学生（张三、李四、王五），你想计算**“Java开发岗 (岗位A)”**和**“Python开发岗 (岗位B)”**的相似度。

我们定义行为分数：投递=5分，收藏=3分，浏览=1分，没看过=0分。

- **岗位 A (Java) 的用户行为向量：**
    - 张三（投递）：5
    - 李四（收藏）：3
    - 王五（没看）：0
    - **向量 A = (5, 3, 0)**
- **岗位 B (Python) 的用户行为向量：**
    - 张三（收藏）：3
    - 李四（浏览）：1
    - 王五（浏览）：1
    - **向量 B = (3, 1, 1)**

**开始套公式代入：**

1. **算分子（内积）：** $(5 \times 3) + (3 \times 1) + (0 \times 1) = 15 + 3 + 0 = 18$
2. **算分母（A的模长 $\times$ B的模长）：**
    - A的模长：$\sqrt{5^2 + 3^2 + 0^2} = \sqrt{25 + 9 + 0} = \sqrt{34} \approx 5.83$
    - B的模长：$\sqrt{3^2 + 1^2 + 1^2} = \sqrt{9 + 1 + 1} = \sqrt{11} \approx 3.32$
    - 分母乘积：$5.83 \times 3.32 \approx 19.35$
3. **得出最终相似度：** $18 / 19.35 \approx 0.93$

**结论：** 0.93 非常接近 1，说明在你的系统里，“Java开发岗”和“Python开发岗”是高度相似的，如果赵六投递了 Java 岗，系统就非常有理由把 Python 岗推荐给他。



## 4. 为什么这个公式最适合你的毕设？

​	在招聘系统中，有些岗位极其热门（大厂岗），有些岗位极其冷门（小微企业）。如果用普通的欧式距离（计算两点之间的绝对距离），冷门岗位和热门岗位永远不可能相似。 但是**余弦相似度不受数值绝对大小的影响**。只要喜欢 A 的那批人，刚好也喜欢 B（比例一致），哪怕 A 有 1000 人投递，B 只有 10 人投递，余弦公式依然能敏锐地发现它们俩是同类。这是写在论文里非常棒的论点。





# 混合推荐

​	在毕设里同时实现**基于物品（Item-based）**和**基于用户（User-based）**的协同过滤，并且在系统中提供这两种推荐策略（做一个混合推荐，赋予这两种推荐计算不同的加权），你的毕业论文直接就有了“对比实验”和“算法优化”的重头戏，这绝对是拿优秀毕设的加分项。

​	基于用户的协同过滤（User-based CF）在逻辑上其实和基于物品非常对称。我们还是用最直白的方式，结合你的招聘系统来拆解。

## 1. 核心思想：它在找什么？

- **基于物品（你之前学的）：** 找“岗位”的相似兄弟。“因为你看了前端岗 A，我给你推荐和 A 很像的前端岗 B”。
- **基于用户（你现在要做的）：** 找“求职者”的相似学长/同学。“因为**张三**和**李四**都投递了腾讯和字节，系统判定他俩是‘相似用户’。现在李四又投递了美团，而张三还没看，系统就把美团推荐给张三。”

**一句话总结：User-based CF 的核心是“人以群分”，利用相似用户的偏好来做推荐。**



## 2. 数学公式：还是那个配方，换了计算方向

底层的数学工具依然是我们刚才讲透的**余弦相似度（Cosine Similarity）**，只是这次我们计算的主体从“岗位向量”变成了“用户向量”。

#### 第一步：计算用户之间的相似度

假设 $u$ 和 $v$ 是两个不同的学生（用户），$I_{u,v}$ 是他们俩都产生过行为（浏览/收藏/投递）的岗位集合。

$$sim(u, v) = \frac{\sum_{i \in I_{u,v}} R_{u,i} R_{v,i}}{\sqrt{\sum_{i \in I_u} R_{u,i}^2} \sqrt{\sum_{i \in I_v} R_{v,i}^2}}$$

- **通俗解释：** 这次我们是在对比数据库里的“行”（用户），而不是“列”（岗位）。如果两个学生经常投递/收藏相同的公司，分子就会很大，他们俩的相似度就越接近 1。

#### 第二步：预测目标用户对某岗位的兴趣得分

我们要预测学生 $u$ 对他没见过的岗位 $i$ 的打分 $P_{u,i}$。我们找出和学生 $u$ 最相似的 $K$ 个学生集合 $S(u,K)$：

$$P_{u,i} = \frac{\sum_{v \in S(u,K)} sim(u, v) \cdot R_{v,i}}{\sum_{v \in S(u,K)} |sim(u, v)|}$$

- **通俗解释：** 找到和张三最像的 5 个同学，看看这 5 个人对“美团前端岗”的打分（比如投递了算 5 分）。把这些分数乘上他们和张三的相似度权重，加起来求平均，就是张三对美团岗位的预测得分。



## 3. 具体实现步骤（代码层面的逻辑）

在你的 Python 后端微服务里，你需要执行以下四个步骤：

1. **构建评分矩阵：** 从 MySQL 读取交互数据，构建一个二维矩阵：行是 `User_ID`，列是 `Job_ID`，里面的值是行为分数（比如投递=5，收藏=3，浏览=1，无行为=0）。
2. **计算用户相似度矩阵：** 利用 `scikit-learn` 的 `cosine_similarity` 函数，直接计算所有用户两两之间的相似度，得到一个对称的 User-User 矩阵。
3. **寻找相似邻居：** 针对当前登录的学生，在矩阵里找到相似度最高的前 $K$ 个用户（比如 Top 10）。
4. **生成推荐列表：**
    - 找出这 10 个相似用户交互过、打过高分的所有岗位。
    - **（关键过滤）** 剔除掉当前登录学生已经投递或看过的岗位。
    - 按照预测得分从高到低排序，把 Top 10 的岗位返回给前端 Vue 进行展示。





## 4. 毕设高分技巧：双剑合璧（混合推荐）

在你的答辩PPT和论文里，我建议你这样设计你的推荐模块，这会显得你的系统非常专业：

- **场景 A：发现新机会（侧重 User-based）**
    - 前端起名：`“与你相似的同学都在投”`
    - 特点：User-based 容易带来“惊喜感”。哪怕是一个完全不同技术栈的岗位，只要相似的同学投了，也会推给你（比如同专业的同学都去考公了，系统也会给你推行政岗）。
- **场景 B：精准匹配（侧重 Item-based）**
    - 前端起名：`“猜你喜欢 / 相似岗位”`
    - 特点：Item-based 非常稳定。你点进一个“Java高级开发”的详情页，下面推荐的清一色都是 Java 相关的岗位。
- **加权混合（终极方案）：**
    - 在最终的“首页推荐流”里，给 User-based 算出的分数乘以 `0.4` 的权重，给 Item-based 的分数乘以 `0.6` 的权重，两者相加得到最终排名。这叫**加权混合推荐（Weighted Hybrid Recommendation）**，写进论文里学术价值极高。
